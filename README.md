Проект содержит теоретические знания по UML диаграммам. Дано описание наиболее используемых в практике UML диаграмм и их графическое представление. 
В проекте содержатся примеры UML  диаграмм, которые можно использовать в работе. 


# Описание  UML  диаграмм
| Название диаграммы           | Что представляет                                                                   | Когда использовать                                                                                                  |
|-----------------------------|-------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------------|
| **Диаграмма классов**       | Описывает типы объектов системы и различного рода статические отношения, которые существуют между ними. На диаграммах классов отображаются также свойства классов, операции классов и ограничения, которые накладываются на связи между объектами                  | Используется для представления иерархии классов и порядка их взаимодействия                                         |
| **Диаграмма последовательности** | Обычно диаграмма последовательности описывает один сценарий. На диаграмме показаны экземпляры объектов и сообщения, которыми обмениваются объекты в рамках одного прецедента (use case).| Диаграммы последовательности следует применять тогда, когда требуется посмотреть на поведение нескольких объектов в рамках одного прецедента. Диаграммы последовательности хороши для представления взаимодействия объектов, но не очень подходят для точного определения поведения.     |
|**Диаграмма компонентов** |  Описывает компоненты ПО и их связи между собой. Например, как микросервисы взаимодействуют друг с другом.| Диаграммы компонентов следует применять, когда система разделяется на компоненты и надо показать их взаимоотношения посредством интерфейсов или схему компонентов в низкоуровневой структуре системы       |
|**Диаграмма прецедентов (USE CASE)**  | Чаще и лучше не рисовать диаграмму прецендентов , а  сделать текстовое описание сценария поведения. <br> Как описывать: <br>  - 1. Определить главный успешный сценарий 2.Добавить расширения (exstension) в главный сценарий. 3.В каждом прецеденте есть ведущий актор 4. Каждый шаг в прецеденте - элемент взаимодействия с системой. 5. Шаг показывает намерение актора, а не механику его действия 6. В прецедент добавляют: Предусловие - описывает действия, обязательно выполняемые системой перед тем, как она разрешит начать работу прецедента. Гарантия  -  описывает обязательные действия системы по окончании работы шаблона ответа. Успешные гарантии выполняются после успешного сценария; минимальные гарантии выполняются после любого сценария. Триггер определяет событие инициирующие выполнение прецедента |Прецеденты представляют собой ценный инструмент для понимания функциональных требований к системе. Первый вариант прецедентов должен составляться на ранней стадии выполнения проекта. Более подробные версии прецедентов должны появляться непосредственно перед реализацией данного прецедента. Важно помнить, что прецеденты представляют взгляд на систему со стороны. |
